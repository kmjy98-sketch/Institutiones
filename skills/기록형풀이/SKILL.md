# 기록형 풀이 스킬 (Claude Code 전용)

사건기록 폴더의 PDF들을 읽고 3단계로 답안을 생성한다. 모든 단계에서 `기록형풀이_룰_2026-05-05.md`의 R1~R10을 준수한다. 특히:
- **R1** 기록 밖 사실 창작 금지 / **R4** 실재 판례만(korean-law-mcp 검증) / **R6** 순환 인용 금지(인용은 항상 원 기록 문서 기준)

## 입력

1. 사건기록 폴더 경로 (예: `7.대회/ai 대회/예시 기록/[Demo] 2026형제8765 상해/`)
   - 신규 기록 표준 위치: `7.대회/ai 대회/기록/{사건번호 사건명}/` — 사건당 폴더 1개에 PDF 일체. 폴더명에 `[대괄호]` 사용 금지(PowerShell 와일드카드 오인).
2. 문제 지시문: 작성할 서면 유형(소장/답변서/참고서면/변호인의견서/검토의견서/공소장)과 당사자 입장. 없으면 **Phase 0 종료 후 사용자에게 확인**.

## Phase 0 — 기록 인벤토리

1. Glob으로 폴더 내 PDF·문서 목록 수집.
2. 파일명 규칙 해석:
   - 민사 전자기록: `법원_사건번호_문서번호_날짜_문서유형_(설명)_제출자.pdf` → 파일명만으로 목록표 1차 구성 가능
   - 형사·기타: 파일명 단순(`고소장.pdf` 등) → 표지의 순번·권·쪽수·증거번호(G번호) 확인
3. 기록 목록표 출력: `| 순번 | 문서명 | 작성자 | 작성일 | 증거번호 |`
4. **일괄 추출 선행(표준, 2026-06-11)**: 목록 수집 직후 pdfplumber 스크립트로 폴더 내 전 PDF를 `sync/대회/_추출/{사건번호}/`에 마크다운으로 일괄 추출한다(스크립트 처리라 토큰 소비 없음). **기존 추출본이 있으면 재추출 금지(#46).** 파일 크기(MB)는 분량 지표로 쓰지 않는다 — 이미지 첨부로 부풀려짐.
5. **면수 기준 분기**: 추출 결과 합계 약 100면 이하 → 전수 정독. 초과 → 인덱스 작성 후 우선순위 정독 + `[자료 부족—보류]` 단정별 보강 문서 매핑 표적 정독.
6. **스캔 PDF 판별**: 추출 텍스트가 빈약하면 스캔본 — 소형은 Read 도구(pages)로 직접 읽기, 대형은 Colab `ocr_extract_v2.ipynb` 경유를 사용자에게 안내.

## Phase 1 — 사실관계 정리

기록 전체를 읽고 다음 5종 산출:

- **A. 타임라인**: 시간순 표, 각 행에 출처 문서·쪽 병기, 다툼 있는 사실 ★ 표시
- **B. 다툼 없는 사실 목록**: 양측 진술이 일치하는 사실 (답안의 전제사실부)
- **C. 당사자 주장 대비표**: 2열 표(원고/검사·피해자 측 | 피고/피의자 측), 빈 셀 `—` (CLAUDE.md #32 형식)
- **D. 증거 분류**: ①서증 ②인적 증거 ③객관 증거(CCTV·포렌식·의무기록·통화기록). 객관 증거와 진술의 일치 여부 메모
- **E. 진술 불일치표**: 같은 사실에 대한 문서 간 차이를 발췌 병기로 대비 → 신빙성·반대신문 포인트

각 항목의 모든 셀에 출처(`《문서명》 p.X`)를 붙인다. 발췌 없이 요약만 적지 않는다.

## Phase 2 — 관련 조문·판례 탐색

1. **쟁점 도출**: Phase 1의 다툼 있는 사실에서 쟁점 번호 매김(최대 6개), 각 쟁점에 잠정 조문 부착.
2. **조문 검증**: 잠정 조문 전부를 korean-law-mcp로 원문 조회 → 원문 발췌 확보. 실패 시 `[검증필요]` + law.go.kr 검색어 제시.
3. **판례 탐색**: 쟁점별로
   - korean-law-mcp 판례 검색 (1차)
   - WebSearch — `site:glaw.scourt.go.kr`, `site:law.go.kr` 한정 (2차)
   - 사용자 보유 자료: `outputs/02_cards/`(주제:: 태그 grep), `sync/wiki/쟁점/` 아티클 (3차 — 단, wiki는 참고일 뿐, 판례 원문 재확인 필수 #40)
   - 확인 실패 시 사건번호 단정 금지, `[검증필요]` + 추천 검색어
4. **결과 정리표**: `| 쟁점 | 조문(원문 발췌) | 판례(사건번호+판시 발췌) | 출처 |`

## Phase 3 — 답안 작성

1. **서면 유형별 골격**:
   - 민사 소장: 청구취지 → 청구원인(요건사실 순) → 입증방법 → 결론
   - 민사 답변서·참고서면: 청구취지에 대한 답변 → 청구원인에 대한 인부(認否는 '인정/부인'으로 표기) → 항변 → 결론
   - 형사 변호인의견서: 사건 개요(공소사실·혐의사실 요지) → 쟁점별 의견[법리→포섭→소결] → 정상관계(양형 사유) → 결론
   - 검토의견서: 질의 → 쟁점별 검토 → 결론 및 조치 의견
2. **IRAC 변형**: 쟁점→법리(Phase 2 결과만 사용)→포섭(Phase 1 발췌 인용, 단 인용 표기는 원 기록 문서)→소결.
3. **상대방 예상 주장 최소 1개 + 반박** 배치. 불리한 쟁점 회피 금지(R9).
4. **문체**: 법원 제출 서면 문체. 분량은 지시문 기준(기본 A4 3매).
5. **생성 후 검증(#13)**: 답안 내 모든 조문번호·사건번호를 korean-law-mcp로 재검증. 검증 실패 단정은 삭제 또는 `[검증필요]`로 강등.

## 산출물 저장

- 위치: 사용자 지정 > 없으면 `sync/대회/{사건번호}/` (예: `sync/대회/2026형제8765/`). 기록 원본 폴더 안에는 생성하지 않는다 (워크스페이스 루트 생성 금지 #42)
- 파일: `01_사실관계정리.md`, `02_쟁점조문판례.md`, `03_답안_{서면유형}.md`
- 추출 캐시: `sync/대회/_추출/{사건번호}/` / 운영 보고서(파일럿·검토류)는 `sync/_meta/` 유지
- 임시 파일은 완료 즉시 `_trash/{YYYY-MM-DD}/`로 이동(#16)
